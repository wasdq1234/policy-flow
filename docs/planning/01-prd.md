## 1. 문제 정의

### 1.1 현재 상황 (As-Is)

정부와 지자체는 매년 수천 건의 지원금을 쏟아내지만, 정보가 '온라인 청년센터', '소상공인시장진흥공단', '기업마당' 등에 흩어져 있다. 마치 보물이 여기저기 숨겨져 있는데 지도가 없는 상황과 같다. 정작 지원이 필요한 사람들이 신청 기한을 놓치거나, 자신에게 해당되는 정책인지조차 모르는 경우가 빈번하다.

### 1.2 목표 상황 (To-Be)

사용자가 자신의 나이, 거주지, 상황만 입력하면, 해당하는 모든 정책이 캘린더 형태로 한눈에 보인다. 마감 임박 시 푸시 알림을 받고, 다른 수혜자들의 꿀팁도 게시판에서 확인할 수 있다. "신청 기간 놓침"이 사라진 세상.

### 1.3 왜 지금인가? (Why Now)

- 공공 데이터 개방이 가속화되어 API 품질이 향상되고 있다
- 경제적 불확실성으로 지원금 정보 수요가 역대급으로 높다
- Cloudflare D1/Workers의 무료 티어가 충분히 성숙해 0원 운영이 가능해졌다

## 2. 사용자 페르소나

### 2.1 주요 페르소나 A: 취업 준비생 김민수

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "달력에 집착하는 민수" |
| 나이/직업 | 26세, 대졸 미취업자 (서울 관악구) |
| 핵심 니즈 | 내 조건에 맞는 지원금만 골라서 일정으로 관리하고 싶다 |
| 불편 포인트 | 카페마다 정보가 다르고, 마감일을 놓쳐서 한숨만 쉰다 |
| 사용 상황 | 아침에 스마트폰으로 이번 주 신청 가능한 정책 확인 |

### 2.2 주요 페르소나 B: 카페 사장 이영희

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "바쁜 영희 사장님" |
| 나이/직업 | 45세, 5년차 개인 카페 운영 (부산) |
| 핵심 니즈 | 키오스크 보조금, 저금리 정책 자금 알림만 받고 싶다 |
| 불편 포인트 | 공문서 투성이라 자격 해당 여부를 모르겠다 |
| 사용 상황 | 가게 쉬는 시간에 알림 확인 후 바로 신청 페이지 이동 |

### 2.3 페르소나의 하루 (시나리오)

**민수의 아침:**
> 9시에 일어나 PolicyFlow 앱을 켠다. "오늘 마감인 정책 1건" 알림이 뜬다. 서울시 청년 월세 지원이다. 캘린더에서 빨간색 바를 탭하니 신청 페이지 링크가 바로 나온다. 게시판에서 "서류 준비 꿀팁" 글을 읽고 점심 전에 신청을 완료한다.

## 3. 사용자 스토리

### FEAT-0: 사용자 설정 (온보딩)

- [x] 사용자로서, 기기 기반으로 자동 로그인되어야 한다. 왜냐하면 매번 로그인하기 귀찮기 때문이다.
- [x] 사용자로서, 나이/거주지/관심분야를 설정할 수 있어야 한다. 왜냐하면 맞춤형 필터링이 핵심이기 때문이다.
- [x] 사용자로서, 소셜 로그인(구글/카카오)을 선택할 수 있어야 한다. 왜냐하면 여러 기기에서 동기화하고 싶기 때문이다.

### FEAT-1: 정책 캘린더 (MVP 핵심)

- [x] 사용자로서, 이번 달 신청 가능한 정책을 캘린더에서 볼 수 있어야 한다. 왜냐하면 한눈에 일정을 파악하고 싶기 때문이다.
- [x] 사용자로서, 지역/분야 필터로 정책을 걸러볼 수 있어야 한다. 왜냐하면 불필요한 정보는 노이즈이기 때문이다.
- [x] 사용자로서, 정책 상세를 눌러 신청 페이지로 바로 이동할 수 있어야 한다. 왜냐하면 클릭 수를 줄이고 싶기 때문이다.
- [x] 사용자로서, 마감 임박 정책이 색상으로 구분되어야 한다. 왜냐하면 시급성을 직관적으로 알고 싶기 때문이다.

### FEAT-2: 북마크 & 알림 (MVP)

- [x] 사용자로서, 관심 정책을 북마크할 수 있어야 한다. 왜냐하면 나중에 다시 확인하고 싶기 때문이다.
- [x] 사용자로서, 북마크한 정책의 마감 3일 전/1일 전 푸시 알림을 받을 수 있어야 한다. 왜냐하면 신청을 놓치고 싶지 않기 때문이다.

### FEAT-3: 커뮤니티 게시판 (MVP)

- [x] 사용자로서, 정책별로 꿀팁/후기/질문을 남길 수 있어야 한다. 왜냐하면 다른 수혜자의 경험이 도움이 되기 때문이다.
- [x] 사용자로서, 익명으로 글을 쓸 수 있어야 한다. 왜냐하면 민감한 정보라 실명은 부담스럽기 때문이다.
- [x] 사용자로서, 댓글과 좋아요로 소통할 수 있어야 한다. 왜냐하면 유용한 정보가 상위로 올라왔으면 좋겠기 때문이다.

### FEAT-4: SEO 정적 페이지 (v2)

- [ ] 검색 유입을 위한 정책별 상세 랜딩 페이지 생성 (Programmatic SEO)

## 5. 범위 관리

### 5.1 MVP (반드시 포함)

- FEAT-0: 사용자 설정 (온보딩, 기기/소셜 로그인)
- FEAT-1: 정책 캘린더 (필터링, 상세, 색상 코딩)
- FEAT-2: 북마크 & 푸시 알림
- FEAT-3: 커뮤니티 게시판 (글/댓글/좋아요)

### 5.2 옵션 (시간 허용 시)

- 모바일 반응형 최적화
- 정책 상세 페이지 내 "비슷한 정책 추천"

### 5.3 Non-goals (명시적 제외)

| 제외 항목 | 이유 |
|----------|------|
| 다크 모드 | MVP 복잡도 최소화 |
| 네이티브 앱 | 웹 푸시로 충분, 앱스토어 심사 리스크 |
| 정책 신청 대행 | 법적 리스크, 단순 정보 제공만 |
| 유료 구독 모델 | MVP 단계에서 수익화 배제 |

## 6. 가정과 리스크

### 6.1 핵심 가정

| # | 가정 | 검증 방법 |
|---|------|----------|
| 1 | 청년센터 API가 안정적으로 동작한다 | Health Check Worker로 일일 모니터링 |
| 2 | 사용자는 푸시 알림을 수락할 것이다 | 가입 후 첫 북마크 시 권한 요청 |
| 3 | 익명 게시판이 악용되지 않을 것이다 | 신고 3회 시 자동 숨김 + Rate Limiting |

### 6.2 Top 5 리스크 보드

| # | 가정 | 리스크 | 완화 | 트리거 | 대응 |
|---|------|--------|------|--------|------|
| 1 | API 안정성 | API 필드 변경/중단 | Health Check + Slack 알림 | 연속 3회 실패 | CSV 파싱으로 폴백 |
| 2 | 무료 티어 한계 | D1 500MB 초과 | 오래된 정책 자동 아카이빙 | 400MB 도달 | 유료 전환 또는 압축 |
| 3 | 스팸 게시글 | 광고/도배 | Rate Limit + 신고 시스템 | 일일 신고 10건 | 관리자 검토 도입 |
| 4 | 날짜 파싱 실패 | 비정형 날짜 30% | "상시" 플래그 처리 | 파싱 실패율 20% 초과 | LLM 도입 검토 |
| 5 | 낮은 사용자 참여 | MAU 목표 미달 | SEO 정적 페이지 우선 | 3개월 후 MAU 1,000 미만 | 마케팅 채널 확대 |

## 7. 해결 가설과 실험

### 7.1 가설

> 만약 정책 정보를 캘린더 형태로 시각화하고 마감 알림을 보내면, 사용자의 정책 신청률이 높아질 것이다. 왜냐하면 현재 가장 큰 문제는 "정보는 있지만 일정 관리가 안 됨"이기 때문이다.

### 7.2 실험 루프

```
MVP 출시 → DAU/MAU 관측 → 북마크/알림 전환율 학습 → v2 기능 우선순위 결정
```

| 단계 | 내용 |
|------|------|
| 실험 | MVP 출시 후 2주간 사용자 행동 추적 |
| 관측 | 북마크 전환율, 푸시 허용률, 게시판 참여율 |
| 학습 | 전환율 15% 이상이면 성공, 미만이면 UX 개선 |
| 다음 | 성공 시 SEO 페이지 확장, 실패 시 온보딩 개선 |
